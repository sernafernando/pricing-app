# Sesión 14/04/2026 — Fixes de estabilidad backend

## Problemas encontrados y resueltos

### 1. Uvicorn se apagaba solo (sin restart)
- **Causa**: `--limit-max-requests 1000` en el `.service` de systemd. Con 4 workers, SSE, background tasks y crons internos, 1000 requests se gastaban en minutos. Cuando el último worker se reciclaba, el parent de uvicorn quedaba en un estado que systemd no detectaba como muerto → no hacía restart.
- **Fix**: Se sacó `--limit-max-requests 1000` del `.service` directamente en el servidor.
- **Estado**: ✅ Resuelto

### 2. Token de MercadoLibre expirado (400 en OAuth)
- **Causa**: El `ml_api_client.py` manejaba su propio flujo OAuth con un `refresh_token` del `.env` que se perdía en cada reinicio. El token ya estaba muerto.
- **Fix**: Reescribí `ml_api_client.py` para leer el `access_token` directamente de la tabla `ml_tokens` en la DB del `ml-webhook` (que ya maneja todo el flujo OAuth). Zero duplicación.
- **Estado**: ✅ Resuelto (requirió re-autenticación en `https://ml-webhook.gaussonline.com.ar/auth`)
- **PRs**: #574

### 3. Pool exhaustion (QueuePool limit reached)
- **Causa raíz**: El endpoint `GET /etiquetas-envio` hacía un `SELECT count(*) FROM (query con 14 LEFT OUTER JOINs con ROW_NUMBER)`. Con un usuario refresheando cada 2-3s, se apilaban 32+ queries idénticas de 10-22 minutos, agotando las 12 conexiones del pool.
- **Fixes aplicados**:
  1. **Count optimizado en listing** (PR #576): Count query separado sin JOINs en el caso común (solo filtro de fecha). De minutos a ~25ms.
  2. **Count optimizado en stats** (PR #579): Misma estrategia para `/etiquetas-envio/estadisticas`. JOINs condicionales solo si los filtros los requieren.
  3. **AbortController en frontend** (PR #576): `TabEnviosFlex` y `EnviosVistaFlag` cancelan requests anteriores al cambiar filtros o refreshear.
  4. **statement_timeout=60s** (PR #578): Red de seguridad — PostgreSQL mata queries que corren más de 60s.
  5. **idle_in_transaction_session_timeout=120s** (PR #578): Mata sesiones zombis que abrieron transacción y nunca hicieron commit/rollback.
- **Nota PgBouncer**: Los timeouts se setean via `@event.listens_for(engine, "checkout")` porque PgBouncer en transaction mode rechaza `connect_args` y `DISCARD ALL` resetea el `SET` entre transacciones. Usar `checkout` (no `connect`) para que aplique en cada uso.
- **Estado**: ✅ Resuelto (monitorear)

### 4. Bug RMA: await de función sync
- **Causa**: `actualizar_caso` y `crear_caso` hacían `await obtener_caso(...)` pero `obtener_caso` es `def` (sync) → `TypeError: object dict can't be used in 'await' expression`
- **Fix**: Sacado el `await` en las 2 llamadas.
- **PRs**: #580
- **Estado**: ✅ Resuelto

### 5. Logs ruidosos
- **Fix parcial**: Bajé 4 logs de búsqueda en `productos_listing.py` de INFO a DEBUG (PR #575).
- **Pendiente**: Los logs de `tienda_nube_order_client` ("Consultando TN API...", "Datos obtenidos...") y `sync_pedidos_preparacion` ("Recibidos N registros", "Tabla truncada") deberían ser DEBUG, no INFO.
- **Estado**: 🔲 Pendiente

## Pendientes / A monitorear

### Pool exhaustion
- El fix cubre el caso común (filtro por fecha sin filtros de JOIN). Si un usuario filtra por `cordon` + `mlstatus` + `ssos_id` simultáneamente con rango de 14 días, las subqueries pesadas se activan. El `statement_timeout=60s` lo mata, pero el usuario ve un error.
- **Posible mejora**: Materializar las subqueries de `soh_status` y `facturado` como columnas en `etiquetas_envio` (desnormalizar). Eso eliminaría los JOINs con ROW_NUMBER por completo.

### Logs a bajar a DEBUG
- `app/services/tienda_nube_order_client.py` — cada request/response individual
- `app/scripts/sync_pedidos_preparacion.py` — "Recibidos N registros", "Tabla truncada", etc.
- `app/services/weather_service.py` — "Weather updated" en cada fetch

### Conexiones idle in transaction
- El `idle_in_transaction_session_timeout=120s` las mata, pero la causa raíz es que hay endpoints async (`async def`) que usan `Depends(get_db)` (sync generator). Cuando el endpoint es async, FastAPI corre el generator en el event loop, y si hay una excepción no manejada, el `finally` del generator puede no ejecutarse.
- **Posible mejora**: Auditar endpoints async que usan `get_db` y migrarlos a sync, o crear un `get_db_async` con `AsyncSession`.

### Dos archivos deps.py
- `app/api/deps.py` (nuevo, con `joinedload` y `get_current_user_transient`)
- `app/core/deps.py` (viejo, sin optimizaciones)
- 27 routers importan de `app.core.deps` — deberían migrar a `app.api.deps`.
